# KF PriceRange — 설계 문서

> 마지막 업데이트: 2026-07-06
> 현재 상태: **3단계 — 파싱+정규화 파이프라인 완성, 전체 데이터 실행 및 그룹핑 구현 예정**

---

## 목표

번개장터 실거래 데이터를 기반으로 "동일한 상품"끼리 묶어서 가격 분포를 보여주는 피처.
SKU가 없는 자유형식 상품명에서 정확하게 동일 상품을 그룹핑하는 것이 핵심 난제.

---

## 대상 카테고리

| 카테고리 | 번장 ID | 설명 |
|---|---|---|
| 스타굿즈 > 보이그룹 > 팬시/포토카드 | `910100001` | 케이팝 포토카드 |
| 키덜트 > 피규어/인형 | `930100` | 애니/포켓몬 등 수집형 피규어 |

두 카테고리는 도메인이 완전히 달라 그룹핑 로직을 별도로 설계.
**현재는 케이팝(910100001) 먼저 진행.**

---

## 그룹핑 키 구조 (케이팝)

동일 상품 = 아래 필드가 모두 일치하는 것

```
아티스트 + 멤버 + 앨범_또는_이벤트 + 소스타입
```

### 필드 정의

| 필드 | 필수 여부 | 설명 | 예시 |
|---|---|---|---|
| `artist` | 필수 | 정규화된 그룹명 | `BTS`, `Stray Kids`, `ENHYPEN` |
| `member` | 필수 | 단일 멤버명 또는 공식 유닛명 (한국어 기준) | `정국`, `니키`, `부석순`, `이오데` |
| `album_or_event` | 셋 중 하나 필수 | 앨범명 / 이벤트명 / 출처명 | `골든`, `로맨스 언톨드`, `콘서트` |
| `source_type` | 필수 | 출처 타입 | Album, Concert, Fan Meeting, ... |

**앨범/이벤트/출처 셋 중 하나는 반드시 있어야 포함.**
멤버명만 있고 나머지 없으면 → 제외.

### source_type 분류

| 값 | 설명 |
|---|---|
| `Album` | 앨범 발매 연계 (정규/미니/싱글/리패키지) |
| `Concert` | 콘서트/투어 연계 |
| `Fan Meeting` | 팬미팅 연계 |
| `Season's Greeting` | 시즌그리팅 |
| `Fan Club` | 팬클럽 가입 혜택 (위버스/공홈 등) |
| `Fan Sign` | 팬사인회 |
| `Collabo` | 브랜드 콜라보 (맥도날드, 메디힐 등) |
| `Benefit` | 구매 혜택 (음반 구매처 특전 등) |
| `Etc` | 위에 해당 없는 공식 출처 |

## 그룹핑 키 구조 (키덜트)

케이팝과 동일하게 4개 필드 복합 키로 그룹핑, 도메인만 다름.

```
프랜차이즈 + 캐릭터 + 제품라인 + 소스타입
```

| 필드 | 필수 여부 | 설명 | 예시 |
|---|---|---|---|
| `franchise` | 필수 | IP/시리즈/브랜드명 (영문 정규화) | `Pokemon`, `One Piece`, `Sanrio` |
| `character` | 캐릭터/제품라인 중 하나 필수 | 특정 캐릭터명 | `Charizard`, `Hello Kitty`, `Luffy` |
| `product_line` | 캐릭터/제품라인 중 하나 필수 | 구체적 발매 시리즈/포맷 | `이치방쿠지 결의 시리즈`, `Nendoroid`, `Bearbrick 48탄` |
| `source_type` | 필수 | 제품 포맷 | Prize Figure, Scale Figure, Nendoroid, Garage Kit, Gacha, Plush, Model Kit, Statue, Etc |

케이팝과 차이점: 캐릭터명이 없어도 제품라인만으로 그룹 가능 (예: 특정 아트토이 시리즈 넘버링).
아직 딕셔너리 기반 정규화 레이어는 없음 — LLM이 프롬프트 지침에 따라 영문 정규 명칭을 직접 출력.

파싱 스크립트: `parse_kidult.py` (`normalize.py`와 별도, `dict/` 미사용)

---

## 제외 규칙

다음 중 하나라도 해당하면 그룹핑 대상에서 제외:

1. **복수 멤버 나열** — "성호 리우 재현 태산 이한 운학" 같이 여러 멤버가 나열된 경우 (단, 공식 유닛명이 있는 조합은 유닛명으로 포함)
2. **일괄/묶음 판매** — 앨범/이벤트가 다른 포카를 한 번에 파는 경우
3. **필수 필드 누락** — 멤버 또는 앨범/이벤트 둘 다 없는 경우
4. **비케이팝** — e스포츠(T1 페이커 등), 포카홀더 같은 액세서리
5. **상품 구별 불가** — 상품명이 너무 불명확한 경우

## 파이프라인 구조

```
[번장 API]
    ↓ 수집 (fetch)
[raw 상품 목록]
    ↓ LLM 파싱 (parse)
[구조화된 필드: artist / member / album_or_event / source_type / exclude]
    ↓ 딕셔너리 정규화 (normalize)
[정규화된 키: BTS / 정국 / 골든 / Album]
    ↓ 그룹핑 (group)
[그룹 단위 상품 묶음]
    ↓ 집계 (aggregate)
[중간값 / 분포 / 최저가 / 거래량]
```

### 각 단계 설명

**1. 수집 (fetch)**
- 번장 API `find_v2.json` 엔드포인트
- `f_category_id`로 카테고리 필터
- 복수 쿼리어로 커버리지 확보
- 중복 pid 제거

**2. LLM 파싱 (parse)**
- Claude Haiku로 배치 처리 (25개씩)
- 상품명 → 구조화 JSON
- 제외 여부도 함께 판단

**3. 딕셔너리 정규화 (normalize)**
- LLM이 파싱한 결과의 표기 불일치 해소
- 예: `ENHYPEN` = `Enhypen` = `엔하이픈` → `ENHYPEN`
- 딕셔너리 파일(`dict/`) 기반으로 매핑

**4. 그룹핑 (group)**
- 정규화된 4개 필드를 복합 키로 사용
- 동일 키 = 동일 상품 그룹

**5. 집계 (aggregate)**
- 그룹별: 중간값, 최저가, 최고가, 거래량, 가격 히스토그램

---

## 딕셔너리 구조 (`dict/`)

정규화에 필요한 사전 파일들. JSON 형태로 관리.

```
dict/
  artists.json      # 그룹명 동의어 사전
  members.json      # 멤버명 동의어 사전
  albums.json       # 앨범명 동의어 사전
  events.json       # 이벤트명 동의어 사전
```

### 형식 예시 (artists.json)

```json
{
  "BTS": ["방탄소년단", "방탄", "BTS", "bts", "bangtan"],
  "Stray Kids": ["스트레이키즈", "스키즈", "Stray Kids", "SKZ"],
  "ENHYPEN": ["엔하이픈", "엔하", "ENHYPEN", "enhypen"],
  "NCT 127": ["엔시티127", "엔시티 127", "NCT127", "NCT 127"],
  "NCT Wish": ["엔시티위시", "엔시티 위시", "NCT Wish", "위시"],
  ...
}
```

| 이슈 | 예시 | 상태 |
|---|---|---|
| 앨범명을 아티스트로 오인 | `타투` → 아티스트로 오인 (실제: NCT 앨범) | LLM이 간헐적으로 재발, 허용 범위 내 |

| 멤버명 없는 상품 | `에픽하이 팬미팅 포카` | 제외 처리로 확정 ✅ |
| 아티스트 인식 실패 시 역추적 | `SMCU 윈윈` → WayV 윈윈 | 역추적 로직 구현됨 ✅ |
